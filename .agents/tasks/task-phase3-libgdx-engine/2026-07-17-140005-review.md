# libGDX view layer with animation-gated state machine (v2)

This change adds a complete libGDX 1.12.1 rendering layer (15 Java files) on top of the existing model/controller architecture. The view connects to `CombatEngine` via `GameEventListener`, renders at 320x240 with a `FitViewport`, and uses an `AnimationManager` queue to prevent the game state machine from racing ahead of visual feedback. All seven issues from the prior review have been addressed: listener deregistration moved to `hide()`, enemy turn guarded against re-entry, skill index mapping replaced with dynamic `PlayerAction.getSkillIndex()` lookup, null guards added to `MessageLog`, viewport unprojection used for click handling, misleading comment corrected, and `CHEST_RESULT` case added.

**Watch for:** The skill index contract between `CombatMenu`, `GameScreen`, and `Player.executeAction()` is now consistent but relies on `PlayerAction.getSkillIndex()` values matching positions in `SkillTree.unlockedSkills` — a contract that's implicit and spread across three files (likely). The skills sub-menu displays Basic Attack as the first entry alongside a dedicated "Attack" option in the main menu, creating redundant paths to the same action (confirmed, minor UX).

**Verdict**: APPROVED

## High-level view

All seven prior concerns are resolved. The listener lifecycle pairs `hide()` deregistration with constructor registration across screen transitions. The enemy turn guard (`processingEnemyTurn` flag) eliminates the re-entry risk that previously depended on synchronous dispatch timing.

The skill index mapping uses a reverse lookup through `PlayerAction.values()` matching on `getSkillIndex()`. The implicit contract is that `PlayerAction.getSkillIndex()` returns the same integer that `SkillTree.getSkillByIndex()` uses — maintained today but spread across `PlayerAction`, `GameScreen`, and `Player` with no single comment documenting the invariant.

## Prior issue resolution

The fixes are implemented as recommended. Key code for reference:

Enemy turn re-entry guard:

```java
case ENEMY_TURN:
    if (!processingEnemyTurn) {
        processingEnemyTurn = true;
        engine.processEnemyTurn();
    }
    break;
```

Skill index dynamic lookup replacing the hardcoded switch:

```java
private PlayerAction getActionForSkillIndex(int skillIndex) {
    for (PlayerAction action : PlayerAction.values()) {
        if (action.getSkillIndex() == skillIndex) {
            return action;
        }
    }
    return null;
}
```

Milestone skills at positions 4+ map to `USE_SKILL_1` through `USE_SKILL_6` via `PlayerAction.getSkillIndex()` returning values 4-9.

## Skill sub-menu UX redundancy

`CombatMenu.renderSkillsMenu()` iterates `player.getSkillTree().getUnlockedSkills()`, which starts with Basic Attack at index 0. The main menu already has "1. Attack" as a dedicated option, so a player navigating to "2. Skills >" sees Basic Attack listed again as the first entry. The sub-menu could skip index 0 or start rendering from index 1 to avoid the duplication.
