# libGDX view layer with animation-gated state machine

This change adds a complete libGDX 1.12.1 rendering layer (15 Java files) on top of the existing model/controller architecture. The view connects to the `CombatEngine` via `GameEventListener`, renders at 320x240 with a `FitViewport`, and uses an `AnimationManager` queue to prevent the game state machine from racing ahead of visual feedback. Screen transitions map 1:1 to `GameState` enum values. No blocking code exists anywhere in the view layer.

**Watch for:** The `CombatMenu` skill-index-to-`PlayerAction` mapping is fragile and will silently pick the wrong action if the `SkillTree` unlocked-skills list isn't ordered the same way GameScreen assumes (confirmed). Screen `dispose()` is never called by libGDX's `Game.setScreen()` so listener deregistration in `GameScreen.dispose()` is dead code, leaking listeners on every screen transition (confirmed). The `processEnemyTurn()` call inside `handleStateTransitions()` executes game logic directly from the render loop without guarding against re-entry on consecutive frames (confirmed).

## High-level view

The rendering pipeline is built around `PixelRenderer` (320x240, `FitViewport`, `OrthographicCamera` with `Nearest` filtering) and a `ScreenShake` component that applies camera offsets based on delta-time falloff. All screens share one `PixelRenderer` and one `AssetLoader` owned by the `Game` subclass, and none allocate GPU resources that need per-screen disposal.

The state machine bridge lives in `GameScreen.handleStateTransitions()`, called every frame after animation updates. It inspects `engine.getCurrentState()` and either issues engine commands (`processEnemyTurn`, `advanceWave`) or swaps screens. The `AnimationManager.isPlaying()` gate at the top prevents advancing state while visual queues are draining, but the `ENEMY_TURN` branch calls `engine.processEnemyTurn()` unconditionally every frame while in that state, which can result in multiple calls before the state changes. In practice, the engine's own state guard and synchronous event dispatch (which sets `isPlaying()` true before the method returns) prevent double-execution today, but this is a fragile implicit contract.

Input is accepted only when the engine is in `AWAITING_PLAYER_ACTION` and no animations are playing. The `CombatMenu` translates number-key or arrow-key selections into a skill index, and `GameScreen.executePlayerAction()` maps that index to a `PlayerAction` enum value via a hardcoded switch. This works for the first four built-in skills but breaks for unlocked skills because `CombatMenu.selectByNumber()` returns a 0-based index into the unlocked skills list, which `GameScreen` interprets as matching `PlayerAction` ordinal positions.

The `MessageLog` implements `GameEventListener` with a delta-time typewriter effect at 60 chars/second and 8-message history. `convertEventToString` calls `getString()` on event data without null guards, producing `"null takes 5 damage"` strings when expected keys are missing.

Click handling in `EventRoomScreen` and `SkillChoiceScreen` uses raw screen-coordinate normalization rather than `viewport.unproject()`, so clicks are misaligned on windows that aren't exactly 4:3.

1. **Listener leak on screen transitions** — `GameScreen.dispose()` unregisters listeners from the engine, but `Game.setScreen()` does not call `dispose()` on the old screen. Move deregistration to `hide()`, or call `dispose()` explicitly before `setScreen()`.
2. **Enemy turn re-entry risk** — `handleStateTransitions()` calls `engine.processEnemyTurn()` every frame while `currentState == ENEMY_TURN`. Currently safe due to the engine's internal state guard plus synchronous event dispatch setting `isPlaying()` true, but a future change to async events or multi-step enemy turns will break this. Guard with a `processingEnemyTurn` flag.
3. **Skill index mismatch** — `CombatMenu.selectByNumber()` returns a 0-based index into `getUnlockedSkills()`, but `GameScreen.getActionForSkillIndex()` maps index 0 to `BASIC_ATTACK`, 1 to `FIREBALL`, etc. An unlocked skill at position 0 in the sub-menu will fire `BASIC_ATTACK` instead. Have `CombatMenu` return the `PlayerAction` or `Skill` directly.
4. **Null event data in MessageLog** — `convertEventToString` calls `event.getString("targetName")` etc. without null checks. Missing keys produce `"null"` strings in the message output. Add fallback values or skip rendering for incomplete events.
5. **Click misalignment on non-4:3 windows** — `EventRoomScreen.touchDown()` and `SkillChoiceScreen.touchDown()` normalize screen coordinates without accounting for FitViewport letterboxing. Use `viewport.unproject(Vector2)` instead.
6. **Misleading comment in DesktopLauncher** — `setBackBufferConfig` configures color/depth/stencil bits, not texture filtering. Nearest filtering is correctly set per-texture in `AssetLoader`; the comment should be removed or corrected.
7. **CHEST_RESULT state not handled in handleStateTransitions()** — The chest UI works via the event listener flag, but a defensive `case CHEST_RESULT: break;` prevents confusion when reading the code and guards against future breakage.

## Listener lifecycle and screen disposal

libGDX's `Game.setScreen(newScreen)` calls `hide()` on the old screen, not `dispose()`. The old `GameScreen` instance becomes garbage with its listeners still registered on the long-lived `CombatEngine`. Every time the game transitions to `SkillChoiceScreen`, `EventRoomScreen`, or `VictoryScreen` and back, a new `GameScreen` is created and two new listeners are added, but the old ones remain. Over a 20-wave run with skill choices and event rooms, dozens of orphaned listeners accumulate, each receiving events and updating stale UI state.

The fix is to move listener deregistration into `hide()`:

```java
@Override
public void hide() {
    engine.removeListener(messageLog);
    engine.removeListener(this);
}
```

## Enemy turn execution from the render loop

In `GameScreen.handleStateTransitions()`:

```java
case ENEMY_TURN:
    engine.processEnemyTurn();
    break;
```

This runs inside `render()` at 60fps. The call fires enemy attack events synchronously via `fireEvent`, which invokes `GameScreen.onEvent()` and queues animations. By the time `processEnemyTurn()` returns, `animationManager.isPlaying()` is true, so the next frame's guard blocks re-entry. The safety relies on two implicit contracts: (1) Java's synchronous listener dispatch guarantees `onEvent` runs before the method returns, and (2) `processEnemyTurn()` always transitions out of `ENEMY_TURN` in a single call. If either assumption breaks (async dispatch, multi-step enemy processing), the enemy attacks multiple times per turn.

## Skill index mapping mismatch

When the player enters the skills sub-menu and selects skill #1, `selectByNumber(1)` returns index `0` (the first unlocked skill). This flows to `GameScreen.executePlayerAction(0)`, which calls `getActionForSkillIndex(0)`, returning `PlayerAction.BASIC_ATTACK`.

The `CombatMenu` returns a 0-based offset into the runtime unlocked skills list. `getActionForSkillIndex` assumes this matches the `PlayerAction` enum ordinal. These two numbering schemes align only if the skill tree's unlocked list is ordered exactly as [BasicAttack, Fireball, HolyLight, IronWill, ...], which happens to be true for the starting skills but breaks the moment an unlocked milestone skill is inserted at a different list position.

## Click coordinate mapping ignores viewport

Both `EventRoomScreen.touchDown()` and `SkillChoiceScreen.touchDown()` convert screen coordinates using:

```java
float normalizedY = 1f - (float) screenY / Gdx.graphics.getHeight();
```

This assumes the viewport fills the entire window. With `FitViewport`, letterbox bars appear when the window aspect ratio isn't 4:3 (the default 960x720 is exactly 4:3, but the window is resizable). On a widescreen monitor maximized, pillarboxing shifts the effective game area, and normalized coordinates no longer map to the correct UI elements. `viewport.unproject()` handles this correctly.
