# MVC decoupling of Plague of Danjin combat engine (v2)

This change extracts a monolithic console RPG into a model/controller split with an event-driven architecture. Game entities live in `src/model/` with zero I/O, an observer system (`src/model/events/`) replaces all `System.out.println`, and a non-blocking state machine (`CombatEngine`) in `src/controller/` orchestrates combat flow. The old flat source files are deleted. The design targets Phase 3 (libGDX integration) where a View layer subscribes to events.

All five issues from the first review have been addressed: mana regen removed from `Player.attack()` (controller owns it), CombatEngine forwards child events as a central bus, `fireEvent()` uses a snapshot array, Random hoisted to fields, and the missing entry point was acknowledged as intentional for Phase 2.

**Watch for:** Stale enemy event forwarding — enemies registered on the central bus are never deregistered after death or wave transitions, which could cause unexpected event propagation in Phase 3 if dead enemy references are held by the view layer (likely).

**Verdict**: APPROVED

## High-level view

The five prior issues are cleanly resolved. `Player.attack()` no longer calls `regenMana()` and carries a Javadoc comment documenting that the controller owns mana timing. CombatEngine uses a `forwardingListener` lambda registered on all child dispatchers (player, enemies, WaveManager, ChestSystem) so a view only needs one registration point on the engine. The `fireEvent()` method snapshots the listener array before iteration. Random instances are fields on Player and Enemy. The engine's `processPlayerAction()` wraps execution in try/catch with state rollback to `AWAITING_PLAYER_ACTION` on failure, eliminating the stuck-state risk.

One new concern surfaces: enemies registered via `registerEnemyListeners()` are never deregistered. The reference direction is `enemy.listeners -> forwardingListener -> engine`, so dead enemies don't prevent GC of themselves (the `currentEnemies` array is reassigned each wave). But if Phase 3 holds references to old enemies for death animations, a stale `fireEvent()` call on a dead enemy would propagate unexpected events to all engine listeners.

## Enemy listener lifecycle and Phase 3 integration risk

The forwarding pattern in CombatEngine:

```java
private final GameEventListener forwardingListener = this::fireEvent;

private void registerEnemyListeners(Enemy[] enemies) {
    for (Enemy enemy : enemies) {
        enemy.addListener(forwardingListener);
    }
}
```

`startNextWave()` creates a fresh `currentEnemies` array and registers listeners on the new enemies, but never calls `removeListener()` on the old ones. The reference direction (`oldEnemy.listeners -> forwardingListener -> engine`) means old enemies don't pin themselves in memory — once `currentEnemies` is reassigned, the old array is eligible for GC.

The actual risk: if Phase 3 code holds references to old enemy objects for death animations or fade-out effects, those objects can still dispatch events through the engine's central bus. A stale animation callback calling `fireEvent()` on a dead enemy would deliver unexpected events to the view layer. Adding a `deregisterEnemyListeners(Enemy[] enemies)` call at the top of `startNextWave()` would close this gap defensively.
