# MVC decoupling of Plague of Danjin combat engine

This change extracts a console RPG game from a single monolithic `GameMain.java` into a proper model/controller split. All game entities move to `src/model/` with zero I/O, an observer-based event system (`src/model/events/`) replaces every `System.out.println`, and a non-blocking state machine (`CombatEngine`) in `src/controller/` orchestrates combat flow. The old flat source files are deleted entirely. The design is oriented toward Phase 3 (libGDX integration) where a View layer would subscribe to events and drive rendering.

**Watch for:** Double mana regen if `Player.attack()` is ever called by the controller instead of `executeAction()` (confirmed). Fragmented event dispatch — five separate `GameEventDispatcher` instances require a listener to register on each independently (confirmed). `ConcurrentModificationException` risk in the listener list during iteration (likely).

## High-level view

The event system uses inheritance (`GameCharacter extends GameEventDispatcher`) to give every character its own listener list. A single view layer needs to register on the player, every enemy spawned per wave, the CombatEngine, the WaveManager, and the ChestSystem — at minimum six registrations for a typical wave. This creates a non-trivial wiring burden for Phase 3.

The CombatEngine operates as a non-blocking state machine with explicit `GameState` transitions. Each method advances exactly one step and returns control to the caller — the correct shape for a frame-driven game loop. However, `PROCESSING_ACTION` is a transient state with no re-entry method; an unexpected exception in `executeAction()` would leave the engine permanently stuck.

Player's mana regen lives in two places: inside `Player.attack()` and inside `CombatEngine.processPlayerAction()`. The engine never calls `hero.attack()`, so no double-regen occurs today. But any future code that calls the player through the `GameCharacter.attack()` interface while using the engine's regen will overshoot by 10 mana per turn.

Game mechanics are faithfully preserved — defense formula, stat caps, enemy scaling, chest probabilities, Lich summon cadence, and GoblinKing rage all match the original byte-for-byte.

<summary>Issues (5)</summary>

1. **Double mana regen contract** — `Player.attack()` calls `regenMana(10)` internally, and `CombatEngine.processPlayerAction()` also calls `hero.regenMana(10)`. If any code path ever reaches `attack()` through the engine, the player gains 20 mana instead of 10. Remove the regen from `Player.attack()` and document that the controller owns mana regen, or remove it from the engine and let `attack()` remain self-contained.
2. **Fragmented event dispatch** — Five separate `GameEventDispatcher` instances (Player, each Enemy, CombatEngine, WaveManager, ChestSystem) require independent listener registration. Add a central event bus or have CombatEngine re-fire child events so a view only registers once.
3. **ConcurrentModificationException in fireEvent** — `GameEventDispatcher.fireEvent()` iterates `listeners` with a for-each loop. If any listener calls `removeListener()` during event processing, the iterator throws. Use `CopyOnWriteArrayList` or iterate over a snapshot.
4. **Random instantiation per call** — `Player.performBasicAttack()` and `Enemy.attack()` create a `new Random()` on every invocation. Wastes allocations and produces poor distribution for rapid successive calls. Hoist to a field.
5. **No GameMain entry point** — The old `GameMain.java` is deleted with no replacement. The project compiles but cannot be run. Phase 3 will need to write the full game loop and view layer from scratch.

## Event system dispatch and listener safety

`GameCharacter extends GameEventDispatcher`, and `GameEventDispatcher` holds a `private final List<GameEventListener> listeners`. Every model object carries its own independent listener list. The controller classes (`CombatEngine`, `WaveManager`, `ChestSystem`) also extend `GameEventDispatcher`.

Enemies are created dynamically by `WaveManager.createWaveEnemies()`, so a view layer must be notified of new enemies and register on them per-wave. This is the primary integration friction point for Phase 3.

The `fireEvent()` loop:

```java
protected void fireEvent(GameEvent event) {
    for (GameEventListener listener : listeners) {
        listener.onEvent(event);
    }
}
```

If a listener removes itself during dispatch (e.g., a one-shot UI animation), `ArrayList`'s iterator will throw `ConcurrentModificationException`. In a libGDX context where UI components may destroy themselves on certain events, this will surface quickly.

## CombatEngine stuck-state risk

The state transitions form this cycle:

```
INTRO -> AWAITING_PLAYER_ACTION -> PROCESSING_ACTION -> ENEMY_TURN
                ^                                          |
                |__________________________________________v
                                                       
WAVE_TRANSITION -> CHEST_RESULT -> (next wave) -> AWAITING_PLAYER_ACTION
                                                       
                -> VICTORY / GAME_OVER (terminal)
```

`processPlayerAction()` sets state to `PROCESSING_ACTION` at entry, then branches to `AWAITING_PLAYER_ACTION` (failure), `ENEMY_TURN` (enemy alive), or `WAVE_TRANSITION` (enemy dead, wave clear). If `executeAction()` throws, the state stays at `PROCESSING_ACTION` — no method accepts that state as an entry condition, so the engine is permanently stuck. Low-risk today (no exceptions in the action logic), but a `try/finally` resetting state on failure would make this defensive.

## Mana regen dual-path semantics

The two call paths produce different mana behavior:

- **Engine path** (`CombatEngine.processPlayerAction`): calls `hero.regenMana(10)` then `hero.executeAction(action, enemy)`. Regen happens once regardless of which action is chosen, including spells.
- **Standalone path** (`Player.attack()`): calls `this.regenMana(10)` then `executeAction(BASIC_ATTACK, target)`. Always regenerates and always basic-attacks.

The engine path matches the original game's behavior (regen happens at start of turn regardless of action choice). The standalone path is a convenience that only makes sense for basic attacks. If anything ever calls `player.attack()` expecting the engine's full action flexibility, it won't get it — it's hardcoded to `BASIC_ATTACK`.

## Phase 3 gaps

The engine has no concept of "wait for animation to finish before proceeding to enemy turn." In libGDX, the view layer will need to gate state transitions on animation completion — e.g., `processEnemyTurn()` should not be called until the player's attack animation finishes. The current API allows calling any transition method at any time, which is correct for a headless engine but requires the view to enforce timing discipline.
