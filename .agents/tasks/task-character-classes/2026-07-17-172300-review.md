# Three-class system with skill trees and save persistence (v3)

All eleven issues from v1 and v2 have been addressed. Arcane Mastery is guarded to apply once (`if (spellCostReductionBonus == 0)`). Immortal Stand cannot be re-armed after trigger (`if (!immortalStandActive)`). `immortalStandActive` is serialized in SaveData and restored via CombatEngineRestorer. The build compiles cleanly.

**Watch for:** Passive skills (Arcane Mastery, Immortal Stand, Plague Bearer) remain in the skill bar as 0-cost/0-cooldown entries after activation; re-casting is a no-op but wastes a turn (confirmed, cosmetic). The `StatusManager.addEffect` replacement path still does not fire `STATUS_EXPIRED`, which would cause a stat leak if buff cooldown were ever reduced below duration in future balance changes (possible, latent only — current cooldown == duration invariant holds).

**Verdict**: APPROVED

## High-level view

The v2 blocking issues are resolved. Arcane Mastery's guard prevents stacking: after the first cast sets `spellCostReductionBonus` to 5, subsequent casts short-circuit. Immortal Stand's guard prevents re-arming: once `immortalStandActive` is true, the skill handler is a no-op regardless of how many times it's cast. Both patterns are preserved across save/load since all three permanent flags (`spellCostReductionBonus`, `plagueBearerActive`, `immortalStandActive`) are serialized in SaveData and restored in CombatEngineRestorer.

The residual risk from `StatusManager.addEffect` replacing same-type effects without firing `STATUS_EXPIRED` remains, but is non-exploitable under the current implementation because Fortress (DEF_BUFF, 3-turn CD, 3-turn duration) and Warlord's Fury (ATK_BUFF, 3-turn CD, 3-turn duration) cannot overlap with themselves. The invariant is implicit rather than enforced, which warrants a code comment but not a ship-blocking fix.

## StatusManager replacement without expiry events

```java
// Non-stackable: remove existing effect of same type
activeEffects.removeIf(e -> e.getType() == effect.getType());
```

CombatEngine reverses ATK_BUFF and DEF_BUFF based on `STATUS_EXPIRED` events containing the potency value. If `addEffect` replaces a buff mid-duration, that potency is never subtracted from the player's stats. Currently impossible because Fortress and Warlord's Fury both have `cooldownTurns == 3` and `duration == 3`, meaning the old buff always expires naturally before the skill is available again. If balance changes reduce either cooldown below 3, the invariant breaks silently. Recommend adding either a comment asserting the invariant, or firing a synthetic expiry event in `addEffect` before the `removeIf`.

</details>
