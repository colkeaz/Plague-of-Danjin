# Three-class system with skill trees and save persistence

This change adds a playable class system (Knight, Mage, Rogue) to the roguelike dungeon crawler. Each class defines starting stats, a passive ability, a 3-skill starting kit, milestone skill choices at waves 5/10/15, class-specific starting equipment, and a distinct pixel art sprite. The game flow now routes through a ClassSelectionScreen between name entry and combat. Save/load persists the chosen class and correctly reconstructs class-specific players on restore.

**Watch for:** Multi-target skills from class trees (Frost Nova, Fan of Knives) will fire as single-target because `isMultiTargetAction` only recognizes `SkillEffect.MULTI_TARGET` (confirmed). TIME_WARP has no engine-side implementation â€” casting it does nothing (confirmed). Immortal Stand and Resurrection share the `autoReviveActive` flag, creating a conflict where one disables the other (confirmed). The Mage's Arcane Affinity cost reduction is bypassed in `processMultiTargetAction`'s mana calculation (confirmed).

## High-level view

Class passives are split across two sites: KEEN_EDGE (crit stats) applies at Player construction, while THICK_SKIN (damage reduction) and ARCANE_AFFINITY (mana regen) are checked at damage/regen time. This split creates a secondary bug: CombatEngine's `processMultiTargetAction` path has its own mana cost calculation that skips both `ARCANE_AFFINITY` and `spellCostReductionBonus` from Arcane Mastery, so Mages pay full price for AoE spells.

The multi-target detection in CombatEngine only recognizes `SkillEffect.MULTI_TARGET`. The class skill trees introduce `STUN_MULTI_TARGET` (Frost Nova) and `POISON_MULTI_TARGET` (Fan of Knives) which will silently fire as single-target skills, hitting only the current enemy.

Several "ultimate" skills have no-op or incorrect implementations. TIME_WARP does nothing. BUFF_DEF_TEMP and SACRIFICE_BUFF are described as temporary but use permanent stat upgrades with no reversal. Permanent runtime flags like `spellCostReductionBonus` and `plagueBearerActive` are not serialized, so those upgrades vanish on save/load.

## Multi-target skills not recognized by CombatEngine

The class skill trees define skills with `STUN_MULTI_TARGET` (Frost Nova â€” "damage + stun all") and `POISON_MULTI_TARGET` (Fan of Knives â€” "damage + POISON all"). CombatEngine only routes to `processMultiTargetAction` when `skill.getSkillEffect() == SkillEffect.MULTI_TARGET`. Frost Nova becomes a strictly inferior Shield Bash (stuns one target for 15 MP vs 10 MP). Fan of Knives becomes a less efficient Poison Strike (same single-target effect for more mana).

The fix: extend `isMultiTargetAction` to recognize the additional multi-target effect types, and extend `processMultiTargetAction` to apply secondary effects (stun/poison) per-enemy after damage.

## TIME_WARP has no engine implementation

```java
case TIME_WARP:
    // Time Warp: the extra turn logic is handled by CombatEngine
    // For now, just mark the state (CombatEngine checks for this)
    break;
```

No state is marked. CombatEngine has zero references to TIME_WARP. The skill (Mage wave 10 milestone, 35 MP, 5-turn cooldown) consumes mana and does nothing. It's offered alongside Meteor (5x damage for 40 MP) and Arcane Mastery (permanent -5 cost). A player choosing Time Warp gets a non-functional ability for the rest of their run.

## Mana cost calculation divergence

Single-target skills go through `Player.executeSkillByIndex()` which applies three cost reductions:
1. `inventory.getTotalStatBonus(SPELL_COST_REDUCTION)` â€” from items
2. `classAbility == ARCANE_AFFINITY` â€” the Mage passive (-3)
3. `spellCostReductionBonus` â€” from Arcane Mastery (-5 permanent)

Multi-target skills go through `CombatEngine.processMultiTargetAction()` which only applies #1. A Mage with Arcane Mastery casting Armageddon (60 MP base) pays 8 MP more than expected. This asymmetry specifically punishes the class whose identity is efficient spellcasting.

## Immortal Stand repurposes autoReviveActive

Immortal Stand's trigger logic:

```java
if (!immortalStandUsed && autoReviveActive && characterClass == CharacterClass.KNIGHT) {
    if (isAlive() && getHp() < getMaxHp() * 0.2) { ... }
}
```

`CombatEngine.isPlayerDeadAfterReviveCheck()` also checks `autoReviveActive` and sets it to false when HP hits 0. If the Knight reaches 0 HP without first passing through the 20% threshold (e.g., one-shot from above 20%), the engine revive fires, consumes `autoReviveActive`, and Immortal Stand's guard condition is permanently false. The two mechanics need independent flags.

## "Temporary" buffs that are permanent

`BUFF_DEF_TEMP` calls `this.upgradeDefense(10)` â€” a permanent stat modification. The Knight's Fortress (15 MP, 3-turn cooldown) reaches the 75 DEF cap in 5 casts. At 75 DEF, all incoming damage is reduced by 75%, making the Knight nearly invulnerable. `SACRIFICE_BUFF` calls `upgradePower(atkBoost)` where atkBoost is 50% of current total ATK â€” permanent and compound-growing (30 -> 45 -> 67 -> 100...).

If these are intentional simplifications, the cooldowns and mana costs should be balanced accordingly. Currently, a Knight can use Fortress 5 times in 15 turns (3-turn CD means it's available every 4th turn) to max DEF while only spending 75 MP total.

## Save/load does not persist permanent skill flags

`spellCostReductionBonus` (from Arcane Mastery â€” "permanently reduce all spell costs by 5") and `plagueBearerActive` (from Plague Bearer â€” "all POISON effects deal double damage permanently") are stored as Player instance fields with no SaveData backing. A Mage who picks Arcane Mastery at wave 10 and saves at wave 12 will lose the -5 cost reduction on reload. Same for a Rogue who picks Plague Bearer at wave 15.

Both skills are wave 10/15 milestone choices with the word "permanently" in their description. The fix is to add these fields to SaveData serialization and restore them in CombatEngineRestorer after Player construction.
