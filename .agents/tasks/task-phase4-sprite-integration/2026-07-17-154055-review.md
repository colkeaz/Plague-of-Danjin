# Programmatic pixel art sprite system for Plague of Danjin

Phase 4 replaces all placeholder colored rectangles with a runtime-generated 8-bit sprite system. SpriteGenerator creates pixel art for 9 entity types (with 4-5 animation states each) plus all UI elements (bars, icons, backgrounds, particles) using libGDX's Pixmap API at startup. A SpriteAnimator state machine drives per-entity animation. VisualEffectsManager layers particles, damage numbers, and screen flash on top of combat. Every screen (Intro, SkillChoice, EventRoom, Victory) gained themed backgrounds, ambient particles, and sprite-based UI.

**Watch for:** Missing `"particle_physical"` texture key (confirmed) disables all particle rendering, damage numbers, screen flash, and HP bar fills at runtime. Damage flash tint on enemy sprite is a no-op (confirmed). The CASTING state loops instead of playing once as documented (confirmed, but not exercised at runtime).

## High-level view

The core design separates generation from animation from effects. SpriteGenerator runs once at startup, produces TextureRegion arrays keyed by entity name and state, and never touches the render loop. AssetLoader acts as a typed facade. SpriteAnimator drives per-entity state transitions (idle loops, attack/hurt play once and return to idle, dying transitions to terminal dead state). Model/ and controller/ remain untouched.

The visual effects layer depends entirely on a particle texture key (`"particle_physical"`) that was never registered. All seven call sites return null, so the entire particle/damage-number/screen-flash system — the primary visual enhancement of this phase — renders nothing at runtime. This is the blocking issue.

All four non-battle screens manage their own particle lists, duplicating VisualEffectsManager's spawn/update/cull pattern. Each screen properly disposes its locally-created Pixmap textures, but these depend on libGDX's screen lifecycle calling `dispose()` — they aren't tracked by SpriteGenerator's shared disposal.

1. **Missing "particle_physical" key** — SpriteGenerator registers `particle_fire`, `particle_holy`, `particle_dark`, `particle_poison`, `particle_impact` but never `particle_physical`. All 7 call sites requesting `getParticleTexture("physical")` return null, disabling particles, damage numbers, screen flash, enemy HP fill, blinking cursor, and selection indicators. 2. **Enemy damage flash is a no-op** — In EnemyDisplay.render(), both the `if (damageFlash)` and `else` branches set `batch.setColor(Color.WHITE)`. The flash was intended to tint the sprite differently (e.g., fully white or red). Change the flash branch to use `new Color(1, 1, 1, 1)` on a flat white draw, or multiply the sprite color.
3. **CASTING loops instead of playing once** — SpriteAnimator.update() groups CASTING with IDLE (both loop indefinitely), contradicting the Javadoc and AnimationState enum doc that says "plays once then returns to IDLE." Not triggered at runtime currently, but will misbehave when spell casting animations are wired up.
4. **Particle lists allocated per-screen without reuse** — IntroScreen, EventRoomScreen, and VictoryScreen each maintain their own `ArrayList<Particle>` with identical spawn/update/cull logic. Consider extracting a lightweight `AmbientParticleEmitter` or reusing VisualEffectsManager to reduce the surface area for particle-related bugs.
5. **getActiveEventType/getActiveEvent are dead code** — AnimationManager exposes two new methods that nothing calls. They were intended as a bridge for effects integration but GameScreen dispatches effects directly in `onEvent` instead. Harmless but adds unused API surface.

## Missing particle texture key

VisualEffectsManager.render() retrieves the default particle texture with:

```java
TextureRegion particleTex = assets.getParticleTexture("physical");
```

This calls `spriteGenerator.getUiSprite("particle_physical")`. SpriteGenerator's `generateParticleTextures()` registers five keys: `particle_fire`, `particle_holy`, `particle_dark`, `particle_poison`, `particle_impact`. There is no `particle_physical` entry. The lookup returns null.

Seven call sites across VisualEffectsManager, GameScreen, IntroScreen, EventRoomScreen, VictoryScreen, SkillChoiceScreen, and EnemyDisplay use this key. All are guarded by null checks, so no crash occurs — but the entire particle and effects layer renders nothing at runtime. This also disables the enemy HP bar fill (EnemyDisplay uses the particle texture as a stretchable fill rect) and the blinking cursor on IntroScreen's name entry.

The fix is a one-line addition in `generateParticleTextures()`:
```java
uiSprites.put("particle_physical", createRegionFromPixmap(pm)); // after generating impact
```

## Damage flash tint no-op

In `EnemyDisplay.render()`:

```java
if (damageFlash) {
    batch.setColor(Color.WHITE);
} else {
    batch.setColor(Color.WHITE);
}
```

Both paths produce the same result. Since `Color.WHITE` is the identity tint for SpriteBatch, the flash does nothing visually. The HURT animation frame still provides some feedback via the SpriteAnimator state change, but the intended white-flash overlay effect is lost.

## CASTING state behavioral inconsistency

SpriteAnimator.update() groups CASTING with IDLE in the looping branch:

```java
case IDLE:
case CASTING:
    // Loop: stateTime wraps around
    break;
```

The AnimationState enum documents CASTING as "plays once then returns to IDLE." When spell casting is eventually wired to trigger `setState(CASTING)`, the animation will loop indefinitely instead of playing once and returning. Fix: move CASTING into the ATTACKING/HURT branch.

## Hardcoded effect positions in GameScreen

GameScreen defines enemy center (160, 170) and player position (50, 45) as static constants, then passes these to VisualEffectsManager for particle/damage-number placement. The HUD places the enemy sprite via EnemyDisplay at `(120, MIDDLE_Y - 40)` and the player mini-sprite at `(70, 30)`. These coordinate sets are not shared between GameScreen and HUD — if the HUD layout shifts, effect positions won't track.

## Screen-local texture disposal

IntroScreen creates `titleTexture` and VictoryScreen creates `victoryPoseTexture` — both are standalone textures not tracked by SpriteGenerator's `managedTextures` list. They depend on libGDX calling `Screen.dispose()` during screen transitions. If a screen is replaced without `dispose()` being called (e.g., via `game.setScreen()` without explicit disposal of the old screen), these textures leak. The PlagueOfDanjinGame screen management code should be verified to ensure it calls `dispose()` on the outgoing screen.
