# Gameplay enhancement layer for Plague of Danjin (Phase 2.5)

This change adds ten interlocking systems on top of the Phase 2 MVC architecture: elemental damage multipliers, status effects with tick processing, an item/equipment system with 19 registry items, a skill tree with milestone unlocks, an enemy telegraph system, four new enemy types (including two mini-bosses), narrative event rooms, run-wide difficulty escalation, an extended event type enum, and full CombatEngine integration. The Phase 2 review findings (fragmented dispatch, stuck-state risk, double mana regen) have all been addressed: the engine now acts as a central event bus with a forwarding listener, `processPlayerAction` wraps logic in try/catch to revert state on failure, and `Player.attack()` no longer calls `regenMana`.

**Watch for:** Six of the nine unlockable skills have incorrect or missing implementations — they are declared with the right names but map to generic `DAMAGE`/`BUFF_DEF`/`BUFF_ATK`/`HEAL` effects that don't match their described behavior (confirmed). BoneColossus shield-break fires a reflect event but never applies the 20 damage to the player (confirmed). Plague Mask poison immunity is declared in `Inventory` but never checked anywhere in the combat pipeline (confirmed).

## High-level view

The elemental system is a static multiplier table on the `Element` enum, integrated into `GameCharacter.takeDamage(int, Element)`. Weapons can override the player's attack element via `Inventory.getElementOverride()`. The multipliers create genuine tactical decisions (Flame Blade against DARK enemies for 2x, same-element halved).

The status effect architecture separates `StatusEffect` (potency/duration), `StatusManager` (tick/expiry lifecycle), and `CombatEngine` (translates tick events into HP changes). The weak spots: ENRAGE is defined but never applied or processed anywhere, and Shield Bash's `STUN` case deals damage without actually applying the STUN status effect.

The skill tree provides four default skills and offers three choices at milestone waves 5, 10, and 15. Only the simplest effects (pure damage, flat +5 DEF, flat +5 ATK, basic heal, multi-target damage) are correctly implemented. Skills with unique mechanics — Drain Life's lifesteal, Assassinate's conditional multiplier, Blood Pact's HP cost, Berserker Rage's temporary stat swing, Divine Shield's multi-hit block, Resurrection's passive revive — all fall through to generic effect handlers that don't reproduce the described behavior.

The enemy telegraph system gives enemies a wind-up turn (firing `ENEMY_TELEGRAPH` with ability name and element), then execution next turn. However, GoblinChieftain, GoblinKing, and Lich all manage telegraph state internally inside their `attack()` override, making the engine's external telegraph logic (`shouldTelegraph`, `windUp`, `getEnemyAbility`) dead code for those enemies — the two parallel schedules are redundant but happen to converge.

Event rooms at waves 3, 7, 13, 17 apply permanent consequences (items, stat changes, run modifier flags). The CombatEngine correctly chains event rooms into the state machine before chest/skill-choice transitions.

## Skill effects mapped to generic handlers

The `Player.applySkillEffect` switch handles six cases: `DAMAGE`, `HEAL`, `BUFF_DEF`, `BUFF_ATK`, `STUN`, and `MULTI_TARGET`. Every unlockable skill is forced into one of these buckets regardless of its intended unique mechanic:

```java
case STUN:
    int stunDamage = (int)(getTotalAttackPower() * skill.getDamageMultiplier());
    target.takeDamage(stunDamage, skill.getElement());
    break;
```

Shield Bash hits the `STUN` case and deals 1.5x damage, but never creates a `StatusEffect(StatusType.STUN, 1, 0, ...)` on the target. The engine already checks `isStunned()` at the top of `processEnemyTurn()` and properly skips the enemy's action — the infrastructure works, the application just isn't wired.

Drain Life (`SkillEffect.DAMAGE`, 2.0x multiplier) should heal the caster for 50% of damage dealt. Blood Pact (`SkillEffect.DAMAGE`, 0 MP cost) should deduct 20 HP from the caster before dealing 5.0x DARK damage. Assassinate (6.0x flat) should check `target.getHp() < target.getMaxHp() * 0.3` and fall back to 2.0x otherwise. These all need either new `SkillEffect` enum values with corresponding logic branches, or a more data-driven approach where `Skill` carries an optional secondary effect descriptor.

The wave 15 skills are the most divergent from spec. Berserker Rage should apply a temporary ENRAGE-like status (+100% ATK, -50% DEF for 3 turns) rather than a permanent flat +5 ATK. Resurrection should set a flag on the player that triggers auto-revival on death, not act as an active 30 HP heal. Divine Shield should apply a SHIELD status with a 2-hit counter rather than a permanent +5 DEF.

## BoneColossus shield reflect is fire-and-forget

When BoneColossus's shield breaks in `takeDamage`:

```java
if (hadShield && !hasShieldNow) {
    shieldActiveThisCycle = false;
    fireEvent(GameEvent.builder(GameEventType.ENEMY_ABILITY_FIRED)
            .put("attackerName", this.getName())
            .put("abilityName", "Shield Reflect")
            .put("reflectDamage", 20)
            .put("element", Element.PHYSICAL.name())
            .build());
}
```

This event is forwarded through the engine's central bus, but `CombatEngine` has no handler that intercepts `ENEMY_ABILITY_FIRED` events and applies the `reflectDamage` to the player. The reflect is purely cosmetic — a view could display "Shield Reflect! 20 damage!" but the player's HP never decreases. The fix requires the engine to register a listener on itself that processes reflect events during the player's action resolution, or have BoneColossus hold a reference to the attacker.

## Poison immunity declared but never enforced

`Inventory.hasPoisonImmunity()` returns `true` when the Plague Mask is equipped. However:

- `PlagueGoblin.attack()` unconditionally calls `target.getStatusManager().addEffect(poison)` without checking the target for immunity.
- `StatusManager.addEffect()` has no concept of immunity.
- `CombatEngine` never intercepts poison application to check equipment.

The correct integration point is either `StatusManager.addEffect()` (accept an immunity set) or the engine's combat loop (check immunity before applying). The current Plague Mask is equivalent to "+3 DEF, +15 MAX_HP" — a stat-stick with a misleading name.

## Dual telegraph schedules in boss enemies

`GoblinChieftain.attack()`, `GoblinKing.attack()`, and `Lich.attack()` all manage telegraph state internally:

```java
// Inside GoblinChieftain.attack()
if (isWindingUp()) {
    executeAbility(target);
} else if (shouldTelegraph(turnCounter)) {
    windUp(WAR_CRY_TELEGRAPH);
    super.attack(target);  // attacks AND telegraphs same turn
} else {
    super.attack(target);
}
```

Meanwhile `CombatEngine.processEnemyTurn()` has its own telegraph logic using `enemyTurnCounts[i]`. When the engine calls `currentEnemy.attack(hero)` on non-telegraph turns, the Chieftain runs its own `shouldTelegraph` check against its own `turnCounter`. The two counters drift independently. On the turns where the engine decides not to telegraph but the enemy's internal counter hits its cadence, the enemy winds itself up inside `attack()`, then on the next engine turn the engine sees `isWindingUp() == true` and calls `executeAbility`.

The net effect is that the engine's telegraph branch (`shouldTelegraph`, `windUp`, `getEnemyAbility`) is dead code for all bosses. Only plain `Enemy` instances (Goblin, Skeleton, PlagueGoblin, ShieldedSkeleton) actually use the engine's telegraph path. This works but is architecturally confusing — two competing mechanisms that happen to not conflict because the engine defers to `attack()` for enemies that self-manage.

## StatusEffect extends GameEventDispatcher needlessly

`StatusEffect.tick()` fires a `STATUS_TICKED` event via its inherited `fireEvent`. Then `StatusManager.tickAll()` also fires a `STATUS_TICKED` event for the same tick. No code ever calls `statusEffect.addListener()`, so the effect-level event fires into an empty listener list every tick. `StatusEffect` inherits `GameEventDispatcher` for no practical purpose, adding an empty `ArrayList` per effect instance.

## ENRAGE status type is dead code

`StatusType.ENRAGE` is documented as "+50% ATK, -25% DEF" but appears nowhere in application logic. Neither `applyPlayerStatusEffects` nor `applyEnemyStatusEffects` in `CombatEngine` handle ENRAGE ticks. No enemy or skill currently applies it. The GoblinChieftain self-buffs via explicit `upgradePower(5)` / `heal(15)` calls instead.
